# Repository Guidelines

## Project Structure & Module Organization
- `src/`: TypeScript source for the CLI and MCP server.
- `src/core/`: service-layer logic (`index-service`, `infra-service`, `health-service`).
- `test/`: Node test suites (`*.test.ts`) plus integration helpers and fixtures.
- `test/fixture` and `test/fixture-realworld`: sample codebases used for parser/indexing tests.
- `indexer/`: optional Python-based indexing flow.
- `dist/`: compiled JavaScript output (generated by `tsc`); CLI entrypoint resolves to `dist/src/cli.js`.
- Root config: `graph-config.json`, `docker-compose.yml`, `tsconfig.json`.

## Build, Test, and Development Commands
- `npm run build`: compile `src/` and `test/` to `dist/`.
- `npm run dev`: TypeScript watch mode for local iteration.
- `npm test`: run Node’s built-in test runner against `dist/test/**/*.test.js`.
- `npm run build && npm test`: recommended full local validation.
- `node dist/src/cli.js infra up --wait`: start Neo4j via project CLI.
- `node dist/src/cli.js index ./test/fixture/src --stats`: run indexer flow against fixture data.

## Coding Style & Naming Conventions
- Language: strict TypeScript (`"strict": true`, NodeNext modules).
- Indentation: 2 spaces; prefer small, single-purpose functions.
- Naming:
  - `camelCase` for variables/functions
  - `PascalCase` for types/interfaces
  - kebab-case for multi-word file names in `src/core/` (for example, `index-service.ts`)
- Use explicit return types on exported functions where practical.
- No dedicated lint/formatter is configured; match existing style and keep imports grouped.

## Testing Guidelines
- Framework: `node:test` with `node:assert/strict`.
- Test files must be named `*.test.ts` under `test/`.
- Tests are executed from compiled output, so always build before testing.
- Prefer deterministic unit tests; for Neo4j-dependent paths, assert meaningful failure messages when DB is unavailable.

## Commit & Pull Request Guidelines
- Follow Conventional Commits seen in history: `feat: ...`, `chore: ...`.
- Include user-story identifiers when applicable (for example, `feat: add index summary output (US-007)`).
- PRs should include:
  - concise problem/solution description
  - linked issue or story
  - test evidence (`npm run build && npm test` output)
  - CLI output examples for command behavior changes

## Security & Configuration Tips
- Do not commit real credentials. Use env vars (`RKG_CONFIG`, `RKG_NEO4J_*`) for local overrides.
- Treat `graph-config.json` defaults as development-only values.
